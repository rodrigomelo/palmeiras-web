# Palmeiras Agenda — Reconcile to VPS + Decommission Vercel

## Goal
Make `workspace/palmeiras-agenda/` the single source of truth, matching the live VPS (`palmeiras.rodrigolanna.com.br`, flat v1.1.37 / app.js v61). Archive the uncommitted monorepo refactor (iOS/Android/services) to a branch. Fix the broken VPS collector. Commit everything to git and decommission Vercel.

## Phase 1 — Move git root to workspace (make workspace the canonical repo)

1. Copy `.git/` from `~/.openclaw/.../palmeiras-web/` into `workspace/palmeiras-agenda/.git/` (so the new root owns the repo + remote `origin → github.com/rodrigomelo/palmeiras-web.git`).
2. Set local git `user.name`/`user.email` (fall back to global config; set if empty).
3. Verify `git status` works from the new location. The openclaw copy becomes a non-git backup (left in place, untouched).

## Phase 2 — Archive the monorepo refactor to a branch (preserve iOS/Android/services)

4. From `workspace/palmeiras-agenda/`, stage **everything** currently there (the monorepo refactor: `apps/ios`, `apps/android`, `apps/web`, `services/`, `packages/`, `infra/`, docs, etc.) and commit on a new branch `monorepo-refactor-wip`.
   - This captures the uncommitted work that currently only lives on local disk.
5. Push the branch to origin (`git push -u origin monorepo-refactor-wip`) so it's safely off-disk. Nothing here is lost.

## Phase 3 — Replace working tree with VPS code (the new source of truth)

6. Back on `main`, wipe the working tree of tracked files (`git rm -rf .` then clean untracked) so the tree is empty.
7. Pull the live VPS code down via `rsync root@177.7.35.32:/var/www/palmeiras-web/ → workspace/palmeiras-agenda/`, **excluding** `.venv/`, `__pycache__/`, `screenshots/`, `.git/`, `.DS_Store`. (Do NOT copy `.env` — keep the local `.env`; do NOT overwrite the `.git/` we just placed.)
   - This brings the flat structure: `server.py`, `index.html`, `static/`, `api/`, `collectors/` (with the real `__init__.py` + `__main__`), `scripts/refresh-all.py`, etc.
8. Restore the `.logs/` dir (used by the launchd jobs) and keep local `.env`.
9. Smoke-test: `PORT=5099 python3 server.py` → verify `GET /` returns 200, `/api/health` runs (503 from local Supabase context is OK).
10. Commit this as `main`: e.g. `chore: reconcile local to VPS production (flat v1.1.37, app.js v61)`.

## Phase 4 — Fix the broken VPS collector (matches/standings currently NOT refreshing)

The deployed systemd unit has two bugs for the flat layout. Fix **both** in `deploy/palmeiras-collector.service`:
- `ExecStart` → `/var/www/palmeiras-web/.venv/bin/python /var/www/palmeiras-web/scripts/refresh-all.py` (the flat orchestrator that exists on the VPS; `services.collector...runner` does not exist there).
- `ReadWritePaths` → `/var/www/palmeiras-web/static/crests /run/palmeiras-collector` (the real crests dir; the `apps/web/` path doesn't exist and crashes systemd namespacing).
Commit the fix locally.
Then deploy + restart on the VPS:
- Copy the corrected unit to `/etc/systemd/system/palmeiras-collector.service`.
- `systemctl daemon-reload && systemctl restart palmeiras-collector.service` and trigger a run.
- Verify: `systemctl status palmeiras-collector.service` shows success, and `/api/v1/health` shows `data_freshness.matches` age dropping on subsequent runs (matches/standings now refreshing again, not just news).

## Phase 5 — Update the local launchd jobs to match the flat structure

11. **`com.palmeiras.web`** (already running) — verify `WorkingDirectory=workspace/palmeiras-agenda`, `python3 server.py` still correct (flat `server.py` is at root). No change expected; reload + confirm HTTP 200.
12. **`com.palmeiras.refresh`** — **change the command back to the flat entry**. The current plist runs `-m services.collector.palmeiras_collector.runner`, but after reconciliation that module won't exist; use the flat `scripts/refresh-all.py`:
    - `ProgramArguments`: `python3 /Users/.../workspace/palmeiras-agenda/scripts/refresh-all.py`
    - Keep `WorkingDirectory=workspace/palmeiras-agenda`, `StartInterval=900` (matches VPS 15-min cadence).
    - Reload + verify one run completes (check `.logs/refresh.log`).
13. Validate all changed plists with `plutil -lint`.

## Phase 6 — Commit + push main; decommission Vercel

14. Ensure `main` is committed and `git push origin main` (brings git + local + VPS into agreement at flat v1.1.37).
15. **Vercel decommission** — *requires auth I don't have* (CLI shows "No existing credentials"). Two options I'll present at execution time:
    - **(a)** You run `vercel login`, then I run `vercel remove palmeiras-web --yes` to delete the project (Vercel stops deploying + serving `palmeiras-web.vercel.app`).
    - **(b)** Manual: you delete the project from the Vercel dashboard.
    - Either way, remove the Vercel git integration so pushes no longer trigger builds.
    - Note: Vercel is currently broken anyway (Supabase disconnected, 6 weeks stale), so this only formalizes its removal. The canonical URL remains `palmeiras.rodrigolanna.com.br`.

## Verification checklist (end state)
- `git log main` HEAD = flat VPS code; `monorepo-refactor-wip` branch preserves iOS/Android/services.
- `git status` clean; `git push` succeeds.
- Local web job: `GET http://127.0.0.1:5001/` → 200.
- Local refresh job: runs `scripts/refresh-all.py`, exits 0.
- VPS collector: `systemctl status` active/success; matches + standings freshness dropping.
- VPS web: `GET https://palmeiras.rodrigolanna.com.br/api/v1/health` → `status: ok`.
- Vercel: project removed (or scheduled for manual removal).

## Risks / safeguards
- **Replacing the working tree destroys local-only files.** Safeguard: Phase 2 commits *everything* to a branch and pushes it off-disk first. Nothing is lost; the refactor is recoverable.
- **VPS collector fix touches a live production service.** Safeguard: minimal unit-file edits (2 lines), `daemon-reload`, one triggered run, verify before leaving it. The web server itself is untouched.
- **`main` will diverge from the old June 3 history forward** — that's intended (the old flat June 3 commit was stale). History is preserved in git log; the refactor preserved on a branch.
- **No data loss to Supabase** — all steps are code/runtime only; the DB is read/written by the running collector, never touched directly by this plan.
